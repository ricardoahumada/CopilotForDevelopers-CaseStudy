# Guía Resumen: GitHub Spec Kit

## Introducción

**GitHub Spec Kit** es un toolkit open source para **Spec-Driven Development (SDD)** que permite a los equipos definir especificaciones estructuradas que se convierten en implementaciones funcionales generadas por agentes de IA. En lugar de prompts vagos, se crean especificaciones que actúan como fuente de verdad para la generación de código.

---

## Instalación

### Prerrequisitos

- Linux / macOS / Windows
- Python 3.11+
- Git
- [uv](https://docs.astral.sh/uv/guides/tools/) (gestor de paquetes Python)
- Un agente de IA compatible (GitHub Copilot, Claude Code, Cursor, Gemini CLI, etc.)

### Instalación Persistente (Recomendada)

```bash
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git
```

### Uso Único (Sin instalación permanente)

```bash
uvx --from git+https://github.com/github/spec-kit.git specify init <NOMBRE_PROYECTO>
```

### Actualización

```bash
uv tool install specify-cli --force --from git+https://github.com/github/spec-kit.git
```

## Comandos CLI

### `specify init`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Inicializa un nuevo proyecto con la estructura de Spec Kit |
| **Para qué se usa** | Crear la estructura de carpetas, templates y configuración inicial del proyecto |
| **Invocación** | `specify init <nombre-proyecto> [opciones]` |
| **Antes** | Tener instalado `uv` y estar en el directorio donde se creará el proyecto |
| **Después** | Se genera la estructura `.specify/` con templates, scripts y memoria del proyecto |

#### Opciones de `specify init`

| Opción | Descripción |
|--------|-------------|
| `<project-name>` | Nombre del directorio del proyecto (usar `.` para el directorio actual) |
| `--ai` | Agente de IA: `copilot`, `claude`, `cursor-agent`, `gemini`, `codex`, `windsurf`, `amp`, `qwen`, `roo`, etc. |
| `--script` | Variante de scripts: `sh` (bash) o `ps` (PowerShell) |
| `--here` | Inicializar en el directorio actual |
| `--force` | Forzar merge/sobrescritura |
| `--no-git` | Omitir inicialización de Git |
| `--ignore-agent-tools` | Omitir verificación de herramientas del agente |
| `--debug` | Habilitar salida de depuración |

### `specify check`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Verifica las herramientas instaladas |
| **Para qué se usa** | Comprobar que el entorno está correctamente configurado |
| **Invocación** | `specify check` |

## Slash Commands (Comandos del Agente)

Estos comandos se ejecutan dentro del agente de IA después de inicializar el proyecto.

### Comandos Principales

#### `/speckit.constitution`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Establece los principios rectores del proyecto |
| **Para qué se usa** | Definir convenciones no negociables: estándares de testing, stack tecnológico, patrones de diseño, políticas de seguridad y compliance |
| **Invocación** | Escribir `/speckit.constitution` en el chat del agente |
| **Antes** | Haber ejecutado `specify init` |
| **Después** | Se crea/actualiza `.specify/memory/constitution.md` |
| **Artefacto generado** | `constitution.md` |

#### `/speckit.specify`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Define los requisitos y user stories del producto |
| **Para qué se usa** | Crear el PRD (Product Requirements Document) - define el **"qué"** y el **"por qué"**, excluyendo decisiones técnicas |
| **Invocación** | `/speckit.specify` seguido de una descripción de alto nivel de lo que se quiere construir |
| **Antes** | Tener definida la constitución del proyecto |
| **Después** | Se genera `spec.md` con escenarios de usuario, requisitos y criterios de aceptación |
| **Artefacto generado** | `specs/<feature>/spec.md` |

#### `/speckit.plan`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Crea el plan técnico de implementación |
| **Para qué se usa** | Definir el **"cómo"**: frameworks, librerías, bases de datos, arquitectura, contratos de datos |
| **Invocación** | `/speckit.plan` proporcionando el stack tecnológico y restricciones |
| **Antes** | Tener el `spec.md` creado |
| **Después** | Se genera `plan.md` con la arquitectura técnica detallada |
| **Artefacto generado** | `specs/<feature>/plan.md` |

#### `/speckit.tasks`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Genera una lista de tareas accionables |
| **Para qué se usa** | Dividir el spec y plan en chunks pequeños, atómicos y paralelizables para implementación |
| **Invocación** | `/speckit.tasks` |
| **Antes** | Tener `spec.md` y `plan.md` creados |
| **Después** | Se genera `tasks.md` con checklist de tareas de implementación |
| **Artefacto generado** | `specs/<feature>/tasks.md` |

#### `/speckit.implement`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Ejecuta todas las tareas de implementación |
| **Para qué se usa** | Que el agente de IA implemente el código basándose en las tareas definidas |
| **Invocación** | `/speckit.implement` |
| **Antes** | Tener `tasks.md` generado y revisado |
| **Después** | El agente genera el código de implementación tarea por tarea |

### Comandos Opcionales

#### `/speckit.clarify`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Clarifica áreas poco especificadas |
| **Para qué se usa** | Identificar y resolver ambigüedades en los artefactos existentes |

#### `/speckit.analyze`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Análisis de consistencia entre artefactos |
| **Para qué se usa** | Verificar que spec, plan y tasks estén alineados y sean coherentes |

#### `/speckit.checklist`

| Aspecto | Descripción |
|---------|-------------|
| **Descripción** | Genera checklists de calidad |
| **Para qué se usa** | Crear listas de verificación para asegurar la calidad del desarrollo |

## Artefactos

### Estructura de Carpetas

```
.specify/
├── memory/
│   ├── constitution.md              # Principios del proyecto
│   └── constitution_update_checklist.md
├── scripts/                         # Scripts de automatización (bash/PowerShell)
├── specs/
│   └── <feature>/
│       ├── spec.md                  # Especificación del producto
│       ├── plan.md                  # Plan técnico
│       ├── tasks.md                 # Lista de tareas
│       ├── research.md              # Investigación (opcional)
│       └── contracts/               # Contratos de datos
└── templates/
    ├── agent-file-template.md
    ├── plan-template.md
    ├── spec-template.md
    └── tasks-template.md

.github/prompts/                     # Prompts para el agente
├── plan.prompt.md
├── specify.prompt.md
└── tasks.prompt.md
```

### Descripción de Artefactos

#### `constitution.md`

| Aspecto | Descripción |
|---------|-------------|
| **Ubicación** | `.specify/memory/constitution.md` |
| **Propósito** | Actúa como el "ADN arquitectónico" del proyecto |
| **Contenido** | Principios no negociables, convenciones de código, stack tecnológico preferido, políticas de testing, estándares de seguridad y compliance |
| **Persistencia** | Se mantiene constante entre features |

#### `spec.md`

| Aspecto | Descripción |
|---------|-------------|
| **Ubicación** | `.specify/specs/<feature>/spec.md` |
| **Propósito** | Documento de requisitos del producto |
| **Contenido** | User journeys, escenarios de uso, requisitos funcionales, criterios de aceptación |
| **Enfoque** | Define el **"qué"** sin entrar en detalles técnicos |

#### `plan.md`

| Aspecto | Descripción |
|---------|-------------|
| **Ubicación** | `.specify/specs/<feature>/plan.md` |
| **Propósito** | Plan técnico de implementación |
| **Contenido** | Arquitectura, tecnologías, frameworks, contratos de datos, guía de quickstart |
| **Enfoque** | Define el **"cómo"** técnicamente |

#### `tasks.md`

| Aspecto | Descripción |
|---------|-------------|
| **Ubicación** | `.specify/specs/<feature>/tasks.md` |
| **Propósito** | Lista de tareas de implementación |
| **Contenido** | Tareas atómicas, ordenadas por fases, con dependencias claras |
| **Formato** | Checklist ejecutable por el agente de IA |

## Casos de Uso Recomendados

1. **Proyectos Greenfield**: Asegura que la IA construya exactamente lo que se pretende desde el inicio
2. **Features en sistemas existentes**: Las specs clarifican cómo las nuevas funcionalidades interactúan con el código existente
3. **Modernización de legacy**: Captura la lógica de negocio en specs modernas y reconstruye sin deuda técnica
